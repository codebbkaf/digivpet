## 2026-07-20 — US-057 — Read an arbitrary health metric over a window
PASSES: true

Did: new Sources/HealthMetricReader.swift (HealthMetricAggregation, ReadableHealthMetric,
HealthReading.aggregate, HealthMetricSampleFetching, HealthKitMetricFetcher, HealthMetricReader)
and new Tests/HealthMetricReaderTests.swift (16 tests). No project.yml edit needed — `sources:
- path: Sources` / `- path: Tests` are whole-directory references; `xcodegen generate` picked
both files up on its own (same as US-056).

Observed (not assumed):
- BUILD SUCCEEDED.
- Full suite 684 tests / 0 failures (was 668 => +16, all mine). Every HealthMetricReaderTests
  case logged "passed" individually — grepped, not inferred.

DESIGN CALLS LATER ITERATIONS MUST NOT UNDO:

1. `ReadableHealthMetric` is a STRUCT with a FAILABLE init from ConditionMetric, not a second
   enum. `init?` returns nil for all five care.* metrics AND for health.sleep. That nil is the
   only gate: HealthMetricReader.read takes a ReadableHealthMetric, so you cannot ask it for a
   care counter or for sleep at all. Do not "helpfully" add a String or ConditionMetric overload
   that falls back to a number.

2. **health.sleep is NOT readable through this reader, on purpose.** Sleep needs the cross-source
   union de-duplication and longest-block rule in SleepAnalysis; counting sleepAnalysis samples as
   events would return a count of SEGMENTS, which is a plausible number and wrong. US-060 must
   route a health.sleep condition to LastNightSleepReader/SleepAnalysis. There is a
   `read(_ metric: ConditionMetric, ...) -> HealthReading?` convenience that returns nil for it
   precisely so the caller is forced to notice.

3. Five aggregations, because one rule would be wrong for most of the vocabulary:
   - sumQuantity(unit)      — steps, distances, flights, exercise/stand/daylight minutes,
                              active/basal kcal, water litres.
   - averageQuantity(unit)  — vo2Max, restingHeartRate, HRV SDNN, respiratoryRate,
                              oxygenSaturation, physicalEffort, audioExposure. These are DISCRETE
                              measurements of a standing state; a stage-long TOTAL resting heart
                              rate is a five-figure number that satisfies any criterion ever
                              written. US-061 must author these as averages.
   - countEvents            — handwashing, toothbrushing, highHeartRate, lowCardioFitness,
                              walkingSteadiness, and workouts (1 HKWorkout = 1 workout).
   - countStoodHours        — appleStandHour ONLY. HealthKit writes an entry for every hour,
                              stood or idle, so countEvents there would report a motionless day
                              as 24. Filters on HKCategoryValueAppleStandHour.stood.
   - sumDurationMinutes     — mindfulSession only. The metric is named health.mindfulMinutes;
                              two 20-minute sessions are 40, not 2.

4. Window rule is IDENTICAL to HealthReading.total: a sample belongs to the window its START falls
   in, half-open [start, end). Kept the same deliberately so a sample counts once and in one
   window whichever reader sees it. testSamplesOutsideTheWindowAreNotCounted pins both edges,
   including that a sample starting exactly at the end instant is OUT.

5. Empty/none-belonging window -> .noData; a window that DID contain samples always yields a real
   .value, including an all-idle stand-hour window, which is .value(0). Fetch throw -> .unavailable,
   never a throw out of read(), matching TodayHealthReader.read.

6. The window is a PARAMETER, not computed inside. US-058 measures over a stage whose start moves,
   and ConditionWindow also has .day and .lifetime — a reader that picked its own window could
   answer only one of the three.

7. testEveryQuantityMetricsUnitIsCompatibleWithItsType is the one test that cannot be replaced by
   a fixture. HKQuantity.doubleValue(for:) raises an ObjC exception on a unit the type does not
   support, and NO fixture test can catch that because fixtures carry a Double, never an
   HKQuantity. It asserts HKQuantityType.is(compatibleWith:) for all 18 quantity entries.
   If you add a metric to the table, that test is what stops a launch-time crash on real hardware.
   Its mirror, testEveryCountedMetricIsACategoryOrWorkoutType, stops the opposite mistake.

8. HealthSample is REUSED for category samples: `value` carries the raw HKCategorySample value
   (only countStoodHours looks at it) and 1 for a workout. Deliberate — SleepSample stays separate
   because sleep's number is its span, and that reasoning is unchanged.

Remains: nothing for this story. NOTHING evaluates conditions yet (still true after US-056);
US-058 (per-stage metric totals) is next, then US-084, US-059, US-060.

Build green. Full suite 702 tests / 0 failures (was 684 => +18, all mine). Every one of the 18
was observed "passed" individually via -only-testing — grepped, not inferred.

DESIGN CALLS LATER ITERATIONS MUST NOT UNDO:

1. **Discrete metrics are NOT accumulated, and this is the important one.** `accumulatesOverTime`
   is false for every `averageQuantity` metric (vo2Max, restingHeartRate, HRV, respiratoryRate,
   oxygenSaturation, physicalEffort, audioExposure). Banking a delta of those would build a
   stage-long TOTAL resting heart rate — a five-figure number that satisfies any criterion ever
   written, which is the exact trap US-057 built `averageQuantity` to avoid.
   => US-060/061: a condition on one of those seven MUST be answered by reading its window
   directly through `HealthMetricReader` (which averages it), never off these totals. There is no
   banked number for them and there must not be. `testOnlyAdditiveMetricsAccumulate` walks the
   whole vocabulary so a metric added later cannot land on the wrong side of the gate silently.
   `care.*` is false too — those counters already live on `GameState`, and a second copy here
   would be free to disagree with the first.
   `health.sleep` IS true: it is not a `ReadableHealthMetric` (it needs SleepAnalysis'
   cross-source de-dup, see US-057 note 2), but the number it yields is MINUTES, and minutes add
   up like any other. Whoever reads the night may bank it here.

2. **THREE totals, not two.** `ConditionWindow` has three cases and `.day` cannot be derived from
   the other two — a stage total says nothing about which single day was the best. Hence
   `stageBestDayMetrics`.

3. **`window: .day` is the BEST day, not today** (the AC, literally). Against today's number a
   good Tuesday would stop counting at Tuesday midnight, and whether a criterion was met would
   depend on what time of day the app happened to be opened. The record tracks the DAY'S running
   total, not the delta each read banked, so three reads of one 12,000-step day are a 12,000-step
   day and not 12,000+9,000+4,000.

4. **Best day is STAGE-scoped and resets on evolution.** Judgement call, defensible either way, so
   here is the reasoning: a `window: .day` condition sits on an evolution EDGE, and edges ask what
   you did to earn THIS evolution. Carried across, one good Tuesday would buy every branch that
   ever asks for a good day, forever. If a later story genuinely needs a lifetime best day, ADD a
   fourth total — do not repurpose this one.

5. **`enterStage(at:)` owns the stage reset**, and is why "stage totals reset when
   `stageEnteredDate` moves" is structurally true rather than true by everyone remembering. Move
   that date anywhere else and you have a bug. A later stage-scoped total gets added in exactly
   one place. NOTE: the debug demo seeders in `MainScreenModel` (lines ~328-525) still assign
   `stageEnteredDate` directly; they are screenshot fixtures, not gameplay, and were left alone.

6. **`enterStage` deliberately does NOT touch the ledger.** Evolving at noon must not re-bank the
   morning's steps for the new form — `testEvolvingMidDayDoesNotRebankTheMorning` pins it. Same
   reason `resetGame`/`rebirth` leave `EnergyLedger` standing.

7. A SECOND ledger rather than fields on `EnergyLedger`: that one is a currency clamped by
   `EnergyRates.dailyCapPerEnergyType`, these are raw uncapped measurements. Folding them together
   would put one day's `min` in the way of the other's total.

8. `noData`/`unavailable` bank nothing and, in particular, cannot overwrite a real best day with a
   zero. Being told nothing is not being told zero — same rule as everywhere else in this codebase.

9. Persisted as plain `[String: Double]?` on the model, with `MetricTotals` as the value-type
   wrapper on top. Optional for the usual migration reason documented all over `GameState`. The
   round-trip test opens a FRESH store; an in-place migration of an already-shipped store is not
   directly tested, but the shape follows the pattern every other added property here used.

REMAINS: nothing for this story. Crediting is NOT yet wired into `MainScreenModel.refresh` — the

1. **The training count is GRADE-BLIND, and this is the one US-075 is most likely to break.**
   `recordTrainingSession()` takes no arguments on purpose — there is nothing to pass it that
   could make a session count for less. DMC counts training whether it succeeded or not, and
   US-061's bands ("8-31 trains the good branch, 32+ overtrains into the junk one") only mean what

   Corollary: do NOT let evolution read `strengthStat` as a proxy for "how much did they train" —

   other.** `refusalCount` is TODAY's and rolls over at local midnight (US-027's mistake is 3+
   refusals in ONE day); `stageOverfeeds` is the stage's and never rolls. A stage-long 3+ overfeed

3. **`stageSleepDisturbances` counts EVERY disturbance; `wakeMistakeDay` still caps the MISTAKE at

2026-07-20 — US-059: Expand HealthKit authorization without breaking the existing grant — PASSES

WHAT WAS DONE
New `Sources/HealthReadSet.swift`. The authorization ask is now a `HealthReadSet` — two halves that
stay separate because they answer different questions:
  - `energyMetrics`: always all four `HealthMetric`. NOT derived from anything. An energy type with
    no grant is a bar that can never fill.
  - `conditionMetrics`: DERIVED from the graph via `EvolutionGraph.conditionHealthMetrics`, which
    walks every node -> edge -> condition, keeps the `health.*` ones and drops `care.*`.
`objectTypes` is the deduplicated union that actually goes to HealthKit; `additionalObjectTypes` is
what a returning user is newly asked about.

`HealthAuthorizing` now takes a `HealthReadSet` instead of `[HealthMetric]` — BOTH methods.
`HealthAuthorizationModel` holds the set (default `.bundled`) and exposes it so the onboarding
screen describes the ask it is about to raise. `HealthOnboardingView` gained a `readSet` parameter
and one conditional line of copy. `NSHealthShareUsageDescription` in project.yml now covers both
purposes.

Tests: `Tests/HealthReadSetTests.swift`, 20 tests. Full suite 737 tests / 0 failures, build green.

DESIGN CALLS LATER ITERATIONS MUST NOT UNDO:

1. **The condition half is DERIVED, and nothing may replace it with a literal list.** The failure
   mode a hardcoded list produces is silent and very expensive to diagnose: HealthKit answers an
   UNAUTHORIZED read with no samples, which is indistinguishable from a user who did not do the
   thing. So a forgotten grant does not error — the edge just never fires, forever, and nothing in
   any log says why. `testAuthoringAConditionIsWhatAddsItsTypeToTheGrant` is the guard: it derives
   the same fixture graph with and without a condition and asserts the type appears. US-061 authors
   ~8+ condition metrics and needs to touch NO authorization code at all; if it finds itself editing
   a list of types, something has been undone.

2. **`ConditionMetric.healthSleep` maps to `.sleepAnalysis` HERE, explicitly, and not via
   `ReadableHealthMetric`.** `ReadableHealthMetric(.healthSleep)` is nil by design (US-057: sleep
   needs the longest-block rule, not a flat total). Leaning on that nil would silently drop sleep
   from the GRANT because it is not readable by that one reader — conflating "how it is read" with
   "whether it is authorized". Today it is granted anyway through `HealthMetric.sleep`, so the bug
   would be invisible until someone changed the energy model.
   `testSleepIsGrantedEvenThoughItsReaderRefusesIt` pins both halves.

3. **The STATUS check covers the wide set, not the four.** This is the whole of AC2 and it is easy
   to get backwards. Narrowed to the four, every existing user's `requestStatus` returns `.answered`,
   `start()` lands on `.ready`, and the new types are never requested — the user is not "re-prompted
   for only the new types", they are never prompted at all. `testTheStatusCheckCoversTheWholeSetNotJustTheFour`.

4. **ONE `requestAuthorization` call for the whole set.** Do not split it into "the old four" plus
   "the new ones" to be polite — that raises two system prompts back to back, and the second one
   gets denied. HealthKit already skips types the user has answered for; re-listing the four in the
   same call cannot revoke or re-ask them. `testTheWholeSetGoesOutInASingleRequest`.

5. `care.*` contributes NOTHING to the ask, and `readObjectType` is nil for it rather than some
   placeholder type. Those counters live in `GameState`.

6. The onboarding copy is a COUNT ("Plus 3 more readings, used only to decide what your Digimon
   evolves into."), not a list. `HealthOnboardingView` fits a 41mm screen only because it is four
   short lines — its own comment explains that burying Continue under a scroll is how a user ends up
   denying access they would have granted, and US-061 will name a couple of dozen metrics. Nil, so
   no line renders at all, when there is nothing extra to ask for.

TESTS: `Tests/HealthAuthorizationTests.swift` still passes, all 15 of them. Every `func test...` BODY

2. **The default context is `.unknown` — a context that answers NOTHING.** A caller that forgets to

    - Unknown reads 0, not "no opinion". A denied or unreadable metric must not warm a hint up, or
      the player is told to keep doing something the app cannot see them doing.
    - An unsatisfied `atMost` is 0, NOT a graded overshoot. Those gates start satisfied and can only
      be broken — every counter behind one (overfeeds, sleep disturbances) grows and never shrinks
      within a stage. Grading the overshoot would show a hint COOLING as the player did more of the
      wrong thing, which reads as progress in the wrong direction.

  is load-bearing: one of them rounding differently than the other would put a checkmark on a branch
  that does not open. `testMetAgreesWithTheEvaluatorAtEveryStep` walks 0→200 and pins it.

  `line(for:in:)` goes through `ConditionHint.resolve`, so US-065's blank/whitespace guard and the
  metric defaults are not bypassed — exactly as the previous iteration's note demanded.

`GameStore.savedState()` (new) — the saved game or nil, WITHOUT creating one. `DexModel` must never
be the thing that hatches a game: opening a side screen before tapping the egg would otherwise save
a Digimon the player never chose. `loadOrCreate` now wraps it.

`DexModel.conditionContext` — built from the saved state in `load()`. `.unknown` on no saved game or
a store that will not open, which degrades every hint to its COLDEST level. That is the right
direction to fail: a hint saying nothing has moved beats a checkmark on something unearned.

  `BattleView.turnDuration`: `sweepDuration` (1.8s round trip), `zoneWidth` (0.25 of the bar),
  `resultDuration` (1.0s), `roundTimeout` (12s).
- A round ALWAYS ends exactly once: the tap's grade, or `.miss` if `roundTimeout` elapses untapped.
  The timeout freezes the marker where it actually was but grades a miss whatever that was — landing
  in the zone by standing still must not pay. `.task(id: stop)` is what hands the grade back.
- The whole screen is the tap target, not the 3pt marker: it is the TIMING that is meant to be hard,
  not the aim. A second tap on the result cannot re-grade it (`guard stop == nil`).

WHY THE BANDS ARE QUARTERS AND HALVES, NOT THIRDS

  perfectShare = 0.25 and greatShare = 0.5 of the zone's HALF-width. Powers of two on purpose. With
  thirds, `zoneWidth/2/3` is not exactly representable, `0.5 + edge - 0.5` rounds by an ulp, and a
  test at the boundary flips on floating-point noise — so "unit-tested at each zone boundary" (AC4)
  would have degraded to "tested a whisker either side of each boundary". With these shares every
  edge is dyadic for any sensible width, and the boundary is asserted ON the line. Edges are
  INCLUSIVE of the better grade: landing exactly on the line is landing in.

2026-07-21 — US-077: Training minigame — Button Masher
================================================================================

WHAT I BUILT

`Sources/ButtonMasherGame.swift` — the second `TrainingMinigame`. Tap as fast as you can; the count
is read against a window and paid out as one of the same four grades. Shape copied from
`TimingBarGame` exactly as its notes advised: pure statics for the rule, stored-prop knobs with

THE ONE DESIGN DECISION WORTH READING: THRESHOLDS ARE A RATE, NOT A COUNT

`tapsPerSecond(for:)` = good 3, great 4.5, perfect 6. `requiredTaps(for:window:)` multiplies by the
window, rounds UP, and floors at 1 for every grade above `.miss`.

Flat counts would have been simpler and wrong. `window` is injectable (AC2), and with fixed
thresholds a halved window silently turns every round into a miss — worse, a test driving a round in
milliseconds could not grade anything at all, so the injectability AC would have bought nothing. As
a rate the game means the same thing at any window: three taps a second is a good round whether it
runs two seconds or ten. At the shipped 5s window the thresholds are 15 / 23 / 30.

The floor at 1 exists because `ceil(6 * 0.001) == 1` only by luck; a literal reading of the
arithmetic at window 0 asks for zero taps, and a perfect handed to someone who never touched the
screen is the one outcome the grading must never produce.

BOUNDARIES WERE FREE HERE, UNLIKE US-076

US-076 spent an afternoon on floating-point band edges and warned the next iteration about it. That
warning does not apply: taps are Int, so "just short of the threshold" is exactly `required - 1` and
every boundary is asserted ON the number with no epsilon anywhere. One table
(`[.good: .miss, .great: .good, .perfect: .great]`) covers all three thresholds and both sides.

THE WINDOW STARTS ON THE FIRST TAP, NOT ON APPEAR

None of the round is then spent realising it has begun; the tap that starts the clock is tap one.
Costs one extra state (`startDate: Date?`) and buys the whole round-lifecycle for free:
`.task(id: startDate)` runs `idleTimeout` while nil and the window itself once set, because setting
`startDate` cancels and re-fires the same task. A round nobody plays still ends, as `.miss`.

THE RULE

`fillRate` = 0.5 meter-fractions/second (whole meter in 2s). Band = [0.625, 0.875]. Below it the
grade still climbs: `great` floor at 0.3125, `good` floor at 0.15625, both shares (1/2, 1/4) of the
band's bottom edge. Above 0.875 the round pays NOTHING — not the grade below, nothing. That cliff is
the game, and it is checked FIRST in `grade(fill:lowerBound:upperBound:)`.

BOUNDARIES WERE FREE AGAIN, AND DELIBERATELY SO

The shipped bounds are eighths and the shipped rate is a half, so every band edge AND every hold
duration derived from one is exactly representable in binary. Boundary assertions are ON the number
and the near-miss is `.nextDown` — the smallest step that exists there — rather than an epsilon
someone picked. US-076 lost an afternoon to float edges; this is the third story in a row to avoid
that by choosing the constants for it. Keep doing this.

`defaultFillRate` / `defaultBandLowerBound` / `defaultBandUpperBound` are named statics that the
stored properties default TO, and the pure tests assert against those names. A later tweak to the
shipped band therefore moves the tests with it instead of leaving them pinning a band nobody plays.

A BUG THE SCREENSHOT CAUGHT THAT THE TESTS COULD NOT

First cut capped the round's fuse at `min(fuse, idleTimeout)`, reasoning that a meter with an
injected rate of zero must still end. That is wrong for any meter slower than the timeout: the
`-powerMeterDemo` run (rate 0.02, fuse 30s) settled itself at 12s on a "Perfect" nobody released,

WHAT I DID

`Sources/CrownSprintGame.swift` (new) — the fourth minigame and the first played with something
other than a finger. Spin the Digital Crown; every unit it travels accumulates toward
`rotationTarget` before `window` closes.

  `.digitalCrownRotation($crown, from: 0, through: 1000, by: 1, sensitivity: .high,
   isContinuous: true, isHapticFeedbackEnabled: true)` on a `.focusable()` / `.focused($focused)`
  screen that takes focus `onAppear` — the crown sends nothing to an unfocused view, and the first
  turn must count rather than be spent waking the view up.

  The binding wanders and WRAPS, so it is never graded directly. `onChange(of: crown)` hands each
  movement to `spun(from:to:)`, which turns it into a distance via
  `crownDelta(from:to:range:)` and adds it to `rotation`.

THE ONE THING THAT WOULD HAVE BEEN A SILENT BUG

`isContinuous: true` means the binding jumps 999 -> 1 as a normal part of spinning. Counted raw
that single wrap is 998 units against a target of 120 — one wrap would win the round outright, and
it would have looked like the game was simply too easy rather than broken. `crownDelta` always
takes the shorter way round (998 -> 2), and `testAWrapCountsAsTheShortWayRoundAndNotTheWholeRange`
pins it, grade included.

Delta is UNSIGNED on purpose: the crown has no forward, and counting one direction only would turn
a speed test into a dexterity test.

REUSE (small, deliberate)

`remainingFraction(at:window:)` moved out of `ButtonMasherGame` into an extension on
`TrainingMinigame`. Same rule, same meaning — how much of a fixed window is left — and the sprint
runs the same draining bar. `ButtonMasherGame.remainingFraction(...)` still resolves through the
protocol extension, so `ButtonMasherTests` is untouched and still green. NOTE the contrast with
`PowerMeterGame.chargeTint`, which the US-078 notes warned not to merge and which stays put: that
one is a live warning about a hold, not a fact about the clock.

TINT: unlike the power meter, the gauge's live tint IS `TrainingResult.tint` of the grade the round
would earn right now. There is no zone to overshoot into here, so live colour and final colour mean
the same thing and there is no second tint rule to keep in step.
