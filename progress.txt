2026-07-21 — US-092: BattleModifiers — effective power from matchup and training
================================================================================
STATUS: PASSES. Build green, 1135 tests / 0 failures (was 1116 — the 19 new ones
are the delta). Pure arithmetic story, no UI, so no Simulator screenshot: every
acceptance criterion is assertable in a unit test and all of them are asserted.

WHAT CHANGED — two NEW files, nothing existing edited
- `Sources/BattleModifiers.swift`
  - `enum BattleModifiers` with the four named constants D-4 asks for:
    `elementAdvantage = 1.25`, `elementDisadvantage = 0.8`,
    `attributeAdvantage = 1.1`, `attributeDisadvantage = 0.9`.
  - `elementFactor(_:against:)` / `attributeFactor(_:against:)` test BOTH
    DIRECTIONS SEPARATELY and multiply both in if both apply. This is the
    load-bearing design choice of the story — see the trap below.
  - `effectivePower(basePower:elementFactor:attributeFactor:trainingFactor:)`
    = `max(1, Int((base * e * a * t).rounded()))`. D-4 verbatim.
  - `matchup(playerPower:playerType:opponentPower:opponentType:training:)`
    returns `BattleMatchup { player, opponent, elementEffectiveness,
    attributeEffectiveness }`, each side a `BattleSideModifiers { basePower,
    elementFactor, attributeFactor, trainingFactor, effectivePower }` plus a
    `totalFactor` convenience. The factors are CARRIED, not multiplied away, so
    US-094 can render the breakdown without re-deriving it (a second
    implementation of D-4 that could drift is exactly what that AC forbids).
    `training` reaches the PLAYER only; the opponent's `trainingFactor` is
    hard-coded 1.0.
  - `TrainingResult.battleMultiplier` in an extension here rather than in
    TrainingMinigame.swift: it is a battle number, and the minigames must not
    grow a dependency on the battle.
- `Tests/BattleModifierTests.swift` — 19 tests, including all five the AC pins:
  light-vs-dark nets 1.0 both ways; even matchup + `good` == raw BattlePower
  exactly; `perfect` + element advantage (child 25 -> 41) beats an adult
  (33 -> 26) AND out-rolls it through `BattleEngine.maximumDamage`; `miss` +
  element disadvantage (25 -> 16) loses to an equal-stage opponent (25 -> 31);
  the opponent's effective power is IDENTICAL across all four grades.

TRAP FOR THE NEXT ITERATION (US-094 will hit this) — effectiveness is read off
the FACTOR, never off `DigimonElement.effectiveness(against:)`
- Those two disagree on exactly one pairing and it is the interesting one.
  `DigimonElement.light.effectiveness(against: .dark)` returns `.advantage`
  (both sides are strong — that type answers "am I strong here?"), but the
  arithmetic is 1.25 x 0.8 = 1.0, a wash. So `BattleMatchup.elementEffectiveness`
  is derived by `BattleModifiers.effectiveness(of: playerElementFactor)` and
  reports `.even`. If US-094 labels the matchup by calling the vocabulary type
  directly it will tell the user "Advantage!" about a fight that got no
  advantage. A test pins both halves of this.
- Same reason `elementFactor` does not `switch` on an `Effectiveness`: a single
  three-case value cannot express "both applied".

BALANCE NOTE, measured rather than assumed
- D-4's prose says a well-played underdog "can beat one rung up and cannot beat
  three". The first half is true and tested. The SECOND HALF IS NOT TRUE of the
  numbers as authored: perfect + element advantage is x1.625 while the opponent
  simultaneously takes x0.8, a 2x swing between the sides, so a child at 25 ->
  41 out-powers an ultimate at 49 -> 39. I did NOT write a test asserting the
  false claim and did NOT retune the constants to make it true — the constants
  are the AC, verbatim. Instead the test asserts what IS true and is the point:
  the best possible boost (1.25 x 1.1 x 1.3 = 1.7875) does not double a Digimon,
  so stage stays the dominant term. If a later story wants the three-rung
  ceiling to hold, the fix is the opponent's disadvantage stacking, not the
  player's boost.
- `BattleEngine` fights on the RATIO of the two powers, so these swings matter
  less than they look: 41 vs 39 is a 4-vs-3 damage ceiling, not a rout.

WHAT REMAINS
- Nothing hooks this up yet. `MainScreenModel.startBattle` still resolves from
  raw `battlePower` — US-093 (pre-battle training round, D-3) is what feeds a
  `TrainingResult` in, and US-094 is what shows the breakdown. `BattleEngine.resolve`

2026-07-21 — US-093: Train before you fight
================================================================================
STATUS: PASSES. Build green, 1145 tests / 0 failures (was 1135 — 9 new tests in
`Tests/PreBattleRoundTests.swift`, plus one added assertion in WanderingTests).
AC8's two screenshots taken from ONE `simctl` launch of `-battleRoundDemo`.

THE SHAPE OF IT — Battle is now two halves, exactly like Train since US-083
- `MainScreenModel.battle()` no longer fights anything. It runs every guard,
  picks the opponent, spends and SAVES the day's allowance, and publishes
  `pendingBattleRound`. It returns `MinigameKind?` now, not `BattleBout?` —
  nil still means "refused", so every existing `XCTAssertNil/NotNil(battle())`
  kept its meaning.
- `finishBattleRound(_ result: TrainingResult) -> BattleBout?` is the other
  half and the one call the minigame's `onFinish` lands in. It builds
  `BattleModifiers.matchup(...)` from the two typings and the grade, resolves
  `BattleEngine` FROM THE EFFECTIVE POWERS, and publishes `pendingBattle`.
- `abandonBattleRound()` is `abandonTraining()`'s twin; `ContentView` calls
  both on `.background`.

THE ONE DESIGN DECISION WORTH KNOWING — the generator is CARRIED
- `PendingBattleRound` holds `{ game, player, opponent, generator }`, and
  `SeededGenerator` gained `Equatable` so it can ride in a `@Published` value.
- Matchmaking happens in `battle()`, BEFORE the allowance is spent, so "Nobody
  to fight." can never cost a battle. The half-used generator is then carried
  into `finishBattleRound`, so the draw sequence across the two halves is
  byte-identical to what the single-call `battle()` used to do. That is what
  kept every pinned-seed test (BattleTests' `seed: 3` loss fixture especially)
  valid without retuning a single seed.
- Had the generator been re-made at grading time, one seed would have produced
  a different fight than before and the outcome-pinned tests would have had to
  be re-derived — which would have hidden any real regression among them.

WHAT THE ROUND COSTS: NOTHING BUT THE ALLOWANCE
- `TrainAction` is never called from `battle()`. No energy, no `strengthStat`,
  no `stageTrainingSessions` — asserted separately, in memory AND read back off
  disk from a reopened store.
- The allowance is spent when the GAME OPENS, not when the grade arrives, and
  flushed immediately. `testTheAllowanceIsSpentAndSavedWhenTheMinigameOpens`
  reopens the store mid-round and finds the battle already charged.

TRAP THAT COST A TEST RUN — hold the reopened `GameStore` in a `let`
- `try GameStore(url: url).loadOrCreate(...)` traps with "This model instance
  was destroyed by calling ModelContext.reset" the moment the returned
  `GameState` is read: the store is a temporary, and it takes its `ModelContext`
  with it when the statement ends. `let reopened = try GameStore(url: url)`
  first. BattleTests got away with it because it holds its store in a `let`.

DEMOS — `-battleRoundDemo` fights as PIYOMON, and that is load-bearing
- Its assigned game is the Reflex Strike, the one of the six that ENDS ON ITS
  OWN: it draws a delay, opens a reaction window, and grades a miss when nothing
  taps it. Agumon's Button Masher sits on "Tap to start" forever, and `simctl`
  cannot tap — the first attempt screenshotted the same waiting round twice.
  With Piyomon, one launch gives the minigame at ~2s and the arena at ~11s with
  NOTHING staged: the real assignment, the real timeout, the real `.miss`.
- Every ARENA demo (`-battleDemo`, `-battleResultDemo`, `-battleTurnDemo`,
  `-battleSignatureDemo`, `-battleLossDemo`, `-battleLimitDemo`) now stages
  `finishBattleRound(.good)` after its `battle()`, the way `-trainResultDemo`
  stages its grade. `good` is the neutral multiplier, so what they screenshot is
  the fight they screenshotted before this story.
- Screenshots need `-healthAnswered` on a fresh install or the authorization
  gate is what lands in the PNG.

WHAT REMAINS / FOR US-094

  breakdown, and it must read the factors off this rather than re-deriving them.
