## Codebase Patterns
- SIMULATOR SCREENSHOTS: `export DEVELOPER_DIR=/Applications/Xcode_26_4_1.app/Contents/Developer`,
  build once against Series 11 (46mm, watchOS 26.4), then `simctl install` the SAME .app onto the
  41mm Series 9 (watchOS 10.4) ‚Äî deployment target is 10.0 and Series 9 is not a usable
  `xcodebuild -destination`. Sim UDIDs: 46mm BA425DD4-8856-4933-8198-0E4B89960E48,
  41mm FB045EDB-DEB9-40AA-B8FB-819E9E742517.
- Launch flags for a screenshot: `-healthAnswered -noNotificationPrompt` plus the demo flag. If a
  notification alert is already queued it SURVIVES terminate+uninstall and eats every shot;
  `simctl erase` on the device is the only fix (then `-healthAnswered` is required or you land on
  onboarding). Cost 10 minutes again this iteration.
- To photograph a pre-story build for a before/after: `git stash push -u`, rebuild, copy the .app
  out of DerivedData FIRST (both builds land on the same path), then `git stash pop`.
- MEASURE screenshots, do not eyeball them. Throwaway CoreGraphics tools compiled with
  `xcrun swiftc` are the pattern (see US-114's /tmp/us114/measure.swift, peak.swift ‚Äî not
  committed, they hard-code this screen's bands). Two that keep earning their keep:
  a "lit in EVERY frame vs lit in ANY frame" pass over a burst separates static chrome from the
  moving sprite, and a peak/mean-luma pass over a band proves a dim is exactly `dimOpacity`.
- The scrim is invisible over a black background, so it cannot be measured directly. Measure what
  it DIMS instead: the sprite's peak luma with the light on vs off.
- NEVER diff a single screenshot per build when the sprite is in shot. The walk1/walk2 alternation
  is driven by a TimelineView that runs whether or not the sprite translates, so two shots taken
  seconds apart routinely catch opposite frames and read as a large bogus diff. Take a BURST of ~6
  per configuration, compare every pair, and use the pair that agrees ‚Äî a matched pair goes to 100%
  byte identity while a mismatched one sticks at the ~50% the two poses share.
- Demo flags that write to the SAVE persist across launches, and several have no inverse
  (`-lightOffDemo` has no `-lightOnDemo`). Order the screenshot pass so the default state is
  photographed FIRST, or `simctl erase` between runs ‚Äî a poisoned shot still looks plausible.
- To prove a constant (an opacity, a scale) is really what the screen paints: render the source
  asset yourself with the same layout math into the measured slot over black, then divide the
  Simulator's band by it. Turns "it looks about right" into a number.
- **THE MAIN SCREEN'S VERTICAL BUDGET IS NOW SPENT ‚Äî genuinely, as of US-120.** `SpriteScale.fitting`
  floors `slotHeight / 16`, and on a hatched Child the slot lands at **49.5pt on 41mm** (a 48pt
  Digimon, **0.5pt of slack**) and **64.0pt on 46mm** (a 64pt Digimon, **0.0pt of slack**).
  **Any row added to `ContentView`'s VStack costs the Digimon a whole scale step, however thin** ‚Äî
  US-120 measured a 13.5pt row taking 48 -> 32pt and 64 -> 48pt before it paid for itself. Measure
  the slot BEFORE designing a row. What is left to pay with, in order of preference:
  - **Nothing cheap.** US-120 already spent the five 1pt VStack gaps, `ActionControls`' 2pt top
    padding, the energy bars' row spacing, and took the name line and the stats values to 9pt.
    `MainScreenTypography` holds those two sizes and a test fails if they grow.
  - The bottom safe area ‚Äî 26pt on 41mm, 36pt on 46mm, real and unused, verified with an
    `.ignoresSafeArea(.container, edges: .bottom)` probe ‚Äî but spending it puts the action row's
    outer circles inside the display's corner arc, which is what the inset is for.
  - The top band holds the clock and BOTH toolbar buttons (US-114's light, the Dex). Not reclaimable.
  - **Price a trade before declining it.** US-120's first iteration set the chrome trim aside as "a
    screen-wide legibility regression"; measured, half of it was pure whitespace costing nothing.
- To measure the sprite SLOT in one command: the map band under `-mapDemo=<asset>` IS the slot,
  exactly, so diff that shot against a `-mapDemo=none` one ‚Äî **but only count a row as slot if
  >=75% of the WIDTH differs.** The map band is full width (344 of 352px on 41mm); `-mapDemo=none`
  also flips the map strip to its prompt state and nudges the name row, so a naive "differs at all"
  diff swallows those two text rows and over-reports the slot by ~23pt. To measure the sprite SCALE:
  a brand-new save is a Digitama whose sheet yields no frames, so `DigimonSpriteView.placeholder`
  draws a stroked box of exactly `16 x scale` points ‚Äî a free ruler that needs no matched walk frame.
  On a HATCHED save use the sprite's ink WIDTH, not its height: height changes with the walk frame.
- **A demo flag's "inverse" is rarely a pure inverse** ‚Äî `-mapDemo=none` undoes US-115's background
  AND changes US-120's strip. When a before/after rests on one flag, check what else that flag
  reaches, or the diff attributes every change to the thing you were testing.
- **Build a pre-story baseline in a `git worktree`, not a `git stash`.** The two builds land in
  different DerivedData directories, so there is no copy-the-app-out-first race and the working tree
  is never disturbed. `git worktree add /tmp/<name> <commit>`, then `git worktree remove --force`.
  A baseline that reproduces the previous iteration's published numbers is what makes a
  before/after worth anything ‚Äî check that it does before trusting the "after".
- `xcodegen generate` is required for a NEW source file even though `project.yml` globs `Sources`.
  The symptom is "cannot find type 'X' in scope" on a file that plainly exists.
- **`Roster` and `EvolutionGraph` are DIFFERENT sets: 1025 entries vs 88 authored nodes.** Data
  authored against the roster (map opponent pools, Digitama slots) CANNOT be resolved through the
  graph ‚Äî it silently comes back empty and the caller falls through to whatever its fallback is.
  US-122 shipped a whole story doing nothing for one build because of this. Until Phase E wires the
  780 orphans, anything meaning "any Digimon" goes through `Roster`. `EvolutionNode` is
  constructible from a `RosterEntry` with an empty `line`, and both `MoveCatalog` and
  `ElementCatalog` degrade gracefully through an empty line ‚Äî that is the bridge (`MapOpponentBand.nodes`).
- **A test that asserts "something came back" cannot catch a silent fallback.** Assert that what
  came back is FROM the set you asked about. In US-122 that one line was the difference between a
  false pass and finding the bug.
- **Anything under `Tests/Fixtures/` is globbed into the test target ‚Äî including files a TOOL leaves
  behind.** Even a read-only `sqlite3 ... "select ..."` on the committed `.store` creates `-wal` and
  `-shm` sidecars, and the next `xcodegen generate` bakes them in; the build then fails with
  `The file "pre-us123.store-shm" couldn't be opened`. Delete the sidecars AND re-run `xcodegen`.
- **`sqlite3` on a store is the way to check what a `@Model` change actually WROTE**, and it is what
  separates "the value read back" from "the value was persisted": a computed accessor with a `nil`
  default returns the same answer either way. `pragma table_info(ZGAMESTATE)` for the columns,
  `select` for the rows. Copy the fixture out first (see above).
- **An optional-backed storage property is the house pattern for adding a field to an
  already-shipped `@Model`**, and the reason belongs on the STORAGE property: SwiftData will not
  migrate a non-optional attribute into an existing store without a default ‚Äî it opens, then the
  next save fails with "is a required value". Choose the `nil` reading in the direction that is SAFE
  for an old save, not merely convenient (US-124: `isActive` reads `nil` as ACTIVE, because the only
  record that can carry one is the single Digimon of a pre-box store).
- **A one-shot migration guarded by "does X exist yet" does not run for a player who already ran the
  build that created X.** US-124's `isActive` stamp lives inside `loadOrCreateProfile`, which
  returns early once there is a profile ‚Äî so it needs the `nil`-default backstop as well. When
  adding a migration, ask which SHIPPED build the player is coming from, not just which schema.
- **NEVER chain a `GameStore` temporary into a `@Model` read in a test:
  `try GameStore(url: url).loadOrCreateProfile()` CRASHES the whole test host.** ARC releases the
  store the instant the call returns, a released store's `ModelContext` is reset, and the returned
  `PlayerProfile`/`GameState` is destroyed with it ‚Äî the next assertion trips
  `SwiftData/BackingData.swift:835: Fatal error: This model instance was destroyed by calling
  ModelContext.reset`. Bind the store to a local first (`let store = try GameStore(url: url)`), and
  keep it alive for as long as anything it handed out is read. Cost a full test run in US-129.
- **A crashed test host makes `xcodebuild test` LIE about the total, and the lie looks like a pass.**
  It relaunches and the final summary counts only the tests since the relaunch ‚Äî US-129 saw
  `Executed 962 tests, with 0 failures` from a suite of 1623, with `** TEST FAILED **` the only
  signal, and that came bundled with an unrelated `xcrun: error: unable to find utility "simctl"`
  from diagnostics collection. ALWAYS compare the summary against
  `grep -rcE '^\s+func test' Tests/*.swift`; if it is short, grep the log for `Fatal error` and
  `Restarting after unexpected exit`. A trailing `simctl` diagnostics error on its own is
  environmental and does NOT mean a test failed ‚Äî read the Executed line.
- **When a story splits "the machinery" from "the data" (US-130/US-131 for Jogress, and every one of
  Phase E's authoring stories), the machinery story's "validator reports zero findings over the
  shipped file" test is VACUOUS ‚Äî say so, and buy back what it cannot prove.** An empty
  `{"recipes": []}` exercises no field name, so a typo in a key survives the story and surfaces as a
  `fatalError` at launch in the next one (every `bundled` here traps by design). Two cheap fixes,
  both used in US-130: decode a LITERAL in the authored shape through the real `JSONDecoder`, and
  leave an explicit `...UntilUS1NN` marker test that FAILS the day the data lands, so the next
  iteration is told rather than having to notice.

  second chance to read them. Renaming is no safer: opening the fixture in `sqlite3` shows SwiftData
  FLATTENS an `EnergyTotals` into one column per field named BY POSITION ‚Äî `ZSTRENGTH1`,
  `ZSTRENGTH2`, `ZSTRENGTH` ‚Äî not by property name, so a rename is a schema change that has to be
  exactly right for the one value nothing is allowed to lose. Four unread columns is the cheaper
  side of that trade.

  parameter and tops it up beside `stageEnergy`; `GameState.battlePower` and `memorial(displayName:)`
  now TAKE the total rather than reading it; `MainScreenModel.lifetimeEnergy` is the single place
  the "no profile yet" answer (`.zero`) is decided.
- Map progress MIGRATES rather than being dropped. `loadOrCreateProfile` adopts `MapProgress`'

  and `rebirth` reads-then-restores it, so both pre-story behaviours hold with the value on its new
  home. A test pins that a rebirth keeps 111/222/333/444 AND the 900 steps already walked.

- **Learnings for future iterations:**
  - **To test a migration, write the fixture BEFORE you write the code.** One throwaway test on the
    current commit, run once, and the store harvested out of
    `~/Library/Developer/CoreSimulator/Devices/<UDID>/data/Containers/Data/Application/<uuid>/tmp/`
    (print the path from the test; it appears in the `xcodebuild` log). It costs one test run and it
    is the only artefact that can prove a migration at all. A `git worktree` build would work too,
    but is not needed if you do it first.
  - **`Tests/Fixtures/*` is bundled into the test target automatically** ‚Äî xcodegen puts unknown
    file types in the resources phase, so no `project.yml` change was needed. Read it with
    `Bundle(for: Self.self).url(forResource:withExtension:)` and COPY IT OUT before opening: SwiftData
    migrates in place, so the second test to run would otherwise find it already migrated.
  - Only the `.store` needs committing. Its `-wal` came out 0 bytes (checkpointed on close) and the
    `-shm` is rebuildable; the tests pass with the two sidecars deleted from the repo.
  - **SwiftData flattens a composite attribute into one column per field, named by POSITION.**
    `EnergyTotals` becomes `ZSTRENGTH/ZVITALITY/‚Ä¶`, `ZSTRENGTH1/‚Ä¶`, `ZSTRENGTH2/‚Ä¶` in declaration
    order. Consequence: removing or reordering one composite is not a local change. `sqlite3
    <store> "pragma table_info(ZGAMESTATE)"` is how to look before leaping.
  - There was already a precedent for schema-migration tests in this repo and it is worth copying:
    `EnergyCreditingTests.testAStoreWrittenBeforeTheLedgerExistedStillOpens` opens a container on an
    explicit OLD `Schema([...])`, and `DominantEnergyTests.PreRecencySchema` declares a whole `@Model`
    of the previous shape inside a private enum. Both work; neither can express a property the new
    build has made private, which is why this story needed a file instead.
  - When a widely-read property moves off a `@Model`, the test churn is in the SETUP, not the
    assertions: 16 files set `state.lifetimeEnergy` to stage a scenario. Three of them
    (`FeedingTests`, `TrainingTests`, `TrainingMinigameTests`) asserted "spending does not rewrite
    the lifetime total" ‚Äî a claim that is now structural, because those actions are handed the
    `GameState` alone. They were kept, on a `PlayerProfile` the action cannot reach, as the test
    that fails the day someone passes the profile in too.

- **Learnings for future iterations:**
  - **THE BUNDLE ID IS `com.digivpet.DigiVPet`, NOT `com.digivpet.app`.** A wrong id fails with
    `FBSOpenApplicationServiceErrorDomain code=4`, which reads like a crash-on-launch and is not.
    `simctl` then happily screenshots the *previous* screen, so the burst looks plausible and is
    stale. Cost ~10 minutes. Read it from the built app:
    `/usr/libexec/PlistBuddy -c "Print :CFBundleIdentifier" "$APP/Info.plist"`.
  - **`simctl erase` REFUSES on a booted device** (`code=405, Unable to erase contents and settings
    in current state: Booted`) and the script carries on regardless. The sequence that works is
    `terminate` ‚Üí `shutdown` ‚Üí `sleep 3` ‚Üí `erase` ‚Üí `boot` ‚Üí `sleep 25-30` ‚Üí `install` ‚Üí `launch`.
    Both sims are usually already booted at the start of an iteration, so this bites immediately.
  - **A ceremony needs a burst with NO sleep between frames.** The existing pattern's `sleep 1` is
    right for a walk cycle but far too slow here ‚Äî 14 back-to-back `simctl io screenshot` calls
    caught the flash; the same count at 1s apart would have landed entirely after it. Then take a
    SECOND, slower burst for the settled screen.

    init nils `reason` when there are offers and substitutes one when there are none, so "the entry
    point never draws an empty list with nothing to explain it" (AC2) is unrepresentable rather than
    tested. Same move as `JogressPair` sorting its ids in US-130.
  - **An offer that CONSUMES what it indexes must be re-derived from the live box before it acts.**
    `performJogress` re-checks the offer against `jogressBoard.offers` and re-matches the recipe on
    the parents' current ids. Taking a Digimon out reorders the box (US-125's thaw moves a birth
    date), so a party screen held across one switch could otherwise fuse two Digimon the player never
    chose ‚Äî worse than `activate(_:)`'s stale-row bug, which only put the wrong one out.
  - **Inject the COMMIT, not just the clock, when a story says "a failure part-way rolls back".**
    There is no other way to make SwiftData throw on demand, and the AC is otherwise unobservable.
    `flush: (() throws -> Void)? = nil`, defaulting to `context.save()`; the app never passes it.
  - **`-jogressListDemo` was added on the US-121 `-mapDetailDemo` precedent.** Whenever a story's new
    UI sits one tap below an existing screen, it is unphotographable without a flag ‚Äî `simctl` has no
    tap command. Budget for the flag as part of the story, not as an afterthought.
