## 2026-07-21 — US-104: The battle projectile vanishes on impact and never flies backward
- Sources/BattleView.swift only. Four changes, two of them found by the Simulator, not by reading:
  1. run()'s reset is now `withTransaction(snap) { projectileProgress = 0; hasLanded = false }` with
     `snap.disablesAnimations = true`. Bare, it shared an update with the animated `beat` change and
     SwiftUI swept it — the 0.15s backward slide the story is named after.
  2. New `@State private var hasLanded`; the projectile draws only while `!hasLanded`. run() sleeps
     `flight`, sets it, then sleeps the rest of the turn, so the ~0.3s tail has NO glyph parked on
     the defender. `flight` = flightDuration clamped to turnDuration, for injected pacings only
     (BattlePacingTests already asserts the shipped pair satisfies it).
  3. `run()` is now @MainActor. Off the main actor its three writes can land in SEPARATE SwiftUI
     updates, and the animated middle one then interpolates the projectile.
  4. `.id(index)` on the projectile, so each exchange's shot is its own view rather than one Image
     reused across turns.
  Fixes 3 and 4 came from a screenshot: at the turn 0 -> 1 boundary, one frame showed AGUMON'S
  ORANGE glyph mid-gap while the HP text was ghosting mid-crossfade — turn 1's shot is Monochromon's
  BROWN one starting at the far right. That is the same bug wearing a different hat, and AC1 says
  "not for a single frame", so it had to go.
- Pure `BattleView.isProjectileVisible(atElapsed:flightDuration:)` — half-open at both ends — is the
  drawing rule as arithmetic, in the `isKnockoutTurn` / `projectileOffset` pattern.
- `BattleView.ProjectileChange` + injected `onProjectileChange`, on the same footing as `playHaptic`:
  SwiftUI animates projectileProgress inside a view no test can mount, so the only observable thing
  is the WRITES — progress, visible, and whether the write is animated.
- Tests: 7 new in Tests/BattleProjectileTests.swift. The AC10 one drives run() through a real
