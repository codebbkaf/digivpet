## 2026-07-17 — US-004: Add SpriteFrame enum and runtime sheet slicing — PASSES

Did:
- Sources/SpriteSheet.swift (one new file, no changes to SpriteLoader or project.yml):
  - `SpriteFrame`: Int-raw enum, walk1=0 ... attack=11. Raw value IS the sheet index, so the
    case order is the sheet layout — do not reorder.
  - `EggFrame`: idle=0, wobble=1, hatch=2 for 48x16 Digitama.

    `kind` is .stage (12 frames) or .egg (3 frames).

    (defaults to SpriteLoader.loadSheet), NSLock-guarded, `.shared` singleton.

Verified (observed, not assumed):
- `xcodebuild build` -> ** BUILD SUCCEEDED **, zero source warnings.
- `xcodebuild test` -> ** TEST SUCCEEDED **, Executed 18 tests (12 new), 0 failures.
- SpriteFrame.attack.sourceRect == (32, 48, 16, 16); walk1/eat1/eat2/hurt1 rects asserted as
  hand-read literals, not by recomputing the formula against itself.
- Child/Agumon -> kind .stage, 12 frames, each 16x16. Digitama/Agu_Digitama -> kind .egg,
  exactly 3 frames.
- Cache: a counting decoder shows 3 sheet() calls for Child/Agumon = 1 decode, and Gabumon
  decodes separately (right key). Second sheet() returns the IDENTICAL CGImage objects (===

DESIGN NOTE — US-005 depends on this:
- EggFrame and SpriteFrame are SEPARATE enums, and each subscript returns nil for the other
  kind of sheet. Egg index 2 is the HATCH but a stage sheet's index 2 is EAT1, so one shared
  enum would make sheet[.eat1] silently return the hatch — the same class of quiet-wrong-art
  bug US-002 found in Bundle.url(forResource:).
  Consequence: `sheet[SpriteFrame.walk1]` on a Digitama is nil BY DESIGN. DigimonSpriteView
  must animate eggs with EggFrame (idle -> wobble) and must NOT fall back to walk frames.
- Failed loads are cached too (cache is [Key: SpriteSheet?]), so a missing sprite does not
  re-hit the disk on every 500ms animation tick.
- Build sprite views on SpriteSheetCache.shared.sheet(stage:name:) — do not call
  SpriteLoader.loadSheet directly from a view, or you re-decode per frame.

DELIBERATE DIFFERENCE FROM US-012 — do not "unify" these two windowing rules:
- Sleep is CLIPPED to its window. US-012's daily quantities are assigned WHOLE to the day they
  START in. The rules differ because the data does: a 900-step sample never says when each step was
  taken, so splitting it would be invention, while a sleep sample's value IS its span and clipping
  it is exact.
- It matters at the 18:00 edge. Someone asleep at 17:30 would otherwise lose the whole night, since
  no other window claims it: consecutive nights do NOT overlap (last night's ends at noon, tonight's

  nap is not last night's sleep. testConsecutiveNightsDoNotOverlap pins the no-double-count half.
- The window is anchored to the local DAY, not to `now`, so it does not drift while the user
  watches. 03:00 reads the night in progress; 23:00 reads the night that ended this morning, not
  the one just beginning.
- DST is covered: the spring-forward night is 17 hours, not 18. Offsets from midnight would run
  past noon into the afternoon the window must never claim.
