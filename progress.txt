## 2026-07-19 — US-036: Add a movement model for horizontal wandering

Added `Sources/Movement.swift` (`MovementModel`) and `Tests/MovementTests.swift` (13 tests).

Design decisions worth not re-deriving:
- Advanced by `advance(to: Date)`, never a timer. Integration is in fixed 0.25s steps
  (`MovementModel.step`) rather than over raw elapsed time, so the path is a function of the seed
  and TOTAL elapsed only — 10x1s lands exactly where 1x10s does. That invariant is what lets
  US-037's TimelineView redraw at any cadence without the walk drifting.
- Part-worn steps are carried, `updatedAt` moves by APPLIED steps only — same reasoning as
  `HungerClock.advance`.
- Absences over `maximumCatchUp` (60s) are DROPPED, not simulated: an unwatched walk did not
  happen, unlike unwatched hunger. Position kept, clock restamped.
- Randomness uses the existing `SeededGenerator` from Battle.swift. 3/8 left, 3/8 right, 2/8 rest;
  4-12 steps per heading. A rest HOLDS facing so a resting sprite does not spin.
- Bound turn uses `>=`/`<=`, not `>`/`<`. Found by a failing test: 30pt/s x 0.25s = 7.5, and
  60/7.5 = 8, so the sprite LANDS exactly on a 60pt bound without overshooting. With `>` it did
  not turn there, sat pinned an extra step, then overshot and turned — a visible stutter that only
  shows up when the bound is a whole multiple of the step distance.
- `bound` is a settable var with a clamping didSet, so US-037 can hand it the real screen width and
  a shrink can never strand the sprite off screen.

Verified: `xcodebuild build` SUCCEEDED. Full suite `Executed 521 tests, with 0 failures`.
project.yml needed no edit — it lists `Sources` and `Tests` as directory paths, so new files are
picked up by `xcodegen generate` alone, which was re-run.

next story and is the first consumer of this model.

## 2026-07-19 — US-037: Render movement in the main screen

Wired US-036's MovementModel into the main screen. New `Sources/WanderingSpriteView.swift`, plus
`offset`/`flipped` on DigimonSpriteView, `isWandering` on MainScreenModel, `hold(at:)` on
MovementModel, and a `-wanderDemo` flag. New tests: Tests/WanderingTests.swift (5), 2 in
MovementTests, 1 in SleepStateTests.

Design decisions worth not re-deriving:
- The pack's art faces LEFT. Confirmed by eye on an upscaled Agumon.png, not assumed. So
  `flipped == (facing == .right)`. Get this backwards and the Digimon moonwalks.
- `.scaleEffect(x: -1)` is applied to the Image AFTER `.interpolation(.none)`, so interpolation is
  what governs sampling on BOTH headings. Verified in the Simulator, not just reasoned about:
  screenshots facing each way are equally crisp. x of exactly -1 is a mirror, so each source pixel
  lands on exactly one destination pixel anyway.
- The sprite moves with `.offset`, which does NOT reserve the ground it covers — that is the point,
  a walking Digimon must not shove the name above it or the caption below it around. The consequence
  is the row must claim `.frame(maxWidth: .infinity)` itself or a sprite at the left bound gets
  clipped by a parent sized to one sprite.
- `SpriteWanderer` is a reference type in @State that publishes NOTHING. The TimelineView is already
  scheduling the redraw, so an ObservableObject would only add "Publishing changes from within view
  updates" warnings for a repaint that was happening regardless. Calling it from `body` is safe
  because `advance(to:)` is idempotent within a step.
- The TimelineView ticks at exactly `MovementModel.step`, so one tick is one step. Faster redraws
  the same position; slower applies several at once and stutters.
- SUSPENSION IS ONE EXPRESSION, NOT A LIST: `animation == .idle && pendingEvolution == nil &&
  pendingBattle == nil && memorial == nil`. Sleeping, sickness and death are ALREADY exactly what
  makes `restingAnimation` return non-idle, so restating them here would be the same fact twice and
  a state added to that switch later would silently keep walking. Eating and every momentary pose
  fall out for free.
- Added `MovementModel.hold(at:)`. A suspension that merely SKIPS advancing leaves `updatedAt`
  behind, and the first advance afterwards applies the whole pause at once — the sprite would
  teleport across the screen the instant it finished a 5s meal. Pauses that short never trip
  `maximumCatchUp`, so that existing guard does not cover this. Two tests pin it, including one
  asserting a held pause does not change the path that follows.
- Added DEBUG-only `-wanderDemo` (healthy Agumon). Necessary, not convenience: every other demo flag
  seeds a state that SUSPENDS movement (sleep/sick/death), and the default save is a Digitama whose
  art is near enough symmetric that a flip is invisible in it. Without it the mirroring AC could not
  be observed at all.

## 2026-07-19 — US-038: Replace the action controls with circular icon buttons

Replaced the three stacked labelled control blocks with one row of four 32x32 circular icon
buttons. New `Sources/ActionControls.swift` (ActionButtonFace + ActionControls); ContentView's

Design decisions worth not re-deriving:
- The circle is an exact `.frame(width:height:)` on a shared `ActionButtonFace`, NOT padding around
  a glyph. Padding would size each button to its own symbol's metrics — fork.knife and bolt.fill
  are not the same width — so the four circles would not match and the cap would be unenforceable.
- `ActionButtonFace` reads `@Environment(\.isEnabled)` rather than taking an `isDisabled` flag. The

  circle that refuses taps.
- `diameter` lives on `ActionButtonFace`, not `ActionControls`. `ActionControls` is generic over its
  bell destination, so `ActionControls.buttonDiameter` fails to infer `Settings` at the use site —
  this cost one build. The face applies the frame, so it is also the honest owner.
- The bell is a `NavigationLink` with a `@ViewBuilder` destination, so ActionControls does not
  import knowledge of NotificationSettingsView and a test can hand it `EmptyView`. That is why the
  type is generic at all. It also puts Notifications IN the row rather than as a labelled link
  below the fold, which is a fourth circle instead of a whole extra line.
- `isBattleDisabled` and `limitCaption` moved across UNCHANGED, both still non-private. The limit is
  the same limit wherever the button is drawn; BattleLimitTests asserts exactly what it did before,
  only constructing an ActionControls. At zero it is still MainScreenModel's OWN refusal string.

  US-039 owns the single stats strip; doing it here would have been two stories in one iteration.
- The four `-*ScrollDemo` flags now all resolve to one anchor. There is no longer a Feed block to

## 2026-07-19 — US-039: Fit the main screen without scrolling

Removed the ScrollView/ScrollViewReader from the playing path. The screen is now one VStack of
fixed-height rows with the sprite as the single flexible one: a GeometryReader measures the slot
the other rows leave and `SpriteScale.fitting(_:)` picks the largest WHOLE scale that fits it.
Whole only — a fractional scale resamples 16x16 art onto a grid it does not line up with, and
`.interpolation(.none)` renders that as uneven pixel widths rather than hiding it as blur.

MEASURE BEFORE YOU BUDGET — this cost most of the iteration. My first cut did only what the AC
literally asked (stats -> one strip, drop the ScrollView) and it BUILT AND RAN, but the screenshot
showed Agumon's head sitting on top of the energy bars. Instrumenting the layout with two throwaway
`print`s gave the real numbers, and they are worth not re-deriving:

- A 42mm screen offers **136pt** of content height to this view (46mm: 150pt). The nav bar with the
  clock and the Dex button eats far more than it looks like it does.
- The fixed rows as they stood cost **123pt** of that 136, leaving the sprite **13pt**.
- The slot is also reported twice per launch and shrinks between passes (35 -> 17.5 on 46mm) as the
  nav bar settles. It converges, and the scale is floor-quantised, so the final value governs — but
  do not trust a single reading.

So the AC's two changes alone could never have fitted a legible Digimon; two more rows had to be
reclaimed, and both are in scope only because the story's actual goal is fitting the screen:

- `EnergyBarsView` went from four stacked rows to a **2x2 grid**: 44pt -> 22pt. The lever that
  mattered was the FONTS, not the bar — a row is as tall as its tallest text, so the 11pt symbol
  was setting the height and the 5pt bar was not. Symbol 11->9, label 9->8, bar 5->4.
- The name's own headline row merged into the caption row as `Name · Stage`, another 16pt. An
  action message still replaces the whole slot, so there is still no layout shift mid-animation;
  what it now costs is the name for the two seconds "Yum!" is up, which is not what the user is
  reading at that moment.

Fixed rows are now ~85pt, so 42mm lands on scale 3 (48pt) with room to spare rather than on the
floor. `SpriteScale.minimum` is **2, not 3** — and this is the actual lesson from the overlap: the
sprite is drawn with `.offset`, so it OVERFLOWS its slot rather than clipping. A floor set above
what the layout can supply does not fail loudly, it silently draws the Digimon over its neighbours.
At 2 it degrades to a small Digimon before anything can collide.

The four `-*ScrollDemo` flags are deleted with the scroll view and are now inert; nothing reads
them. `MainScreenModel`'s `-battleLimitDemo` doc no longer tells the reader to pair one.

## 2026-07-19 — US-040: Add line metadata to the evolution graph

Added a required `line` field to `EvolutionNode` and tagged all 22 shipped nodes: agumon (8),
gabumon (7), palmon (7). New `GraphValidationError.emptyLine`; docs/evolutions-schema.md updated.
No evolution logic reads `line` — it is purely the grouping key US-041's tree view will need.

Two decisions worth not re-deriving:

- **`line` is `decode`, not `decodeIfPresent`.** Every other optional-ish node field (`dexOnly`,
  `evolutions`, `variant`) defaults, so defaulting this one would have matched the file's habit —
  but the failure mode is the argument against it. A node that quietly joined a fallback line
  decodes fine, then goes missing from the Dex tree a user looks for it in, and the symptom
  (absent art, one screen away) names neither the node nor the JSON. Failing the whole load puts
  the error on the line that caused it. The validator's `emptyLine` rule therefore only ever
  catches `""` — a MISSING key cannot reach validation at all.

- **The memberwise init defaults `line` to `"test"`; the decoder does not.** There are 83
  `EvolutionNode(...)` sites in Tests/ and none in Sources/ (verified by grep), essentially all of
  them building a node to exercise something unrelated to grouping. Threading a real line through
  all 83 would have been churn that made every fixture read as if grouping mattered to it. The
  default is unreachable from shipped code, so it cannot mask a real missing line.

Meramon is tagged `agumon`, not a line of its own: it is the target of the one branching edge and
converges back at MetalGreymon, so it belongs in the Agumon tree as a branch. US-041 will draw it
that way. `testBundledNodesAreTaggedWithTheirLine` asserts the exact per-line COUNTS rather than
just non-emptiness, which is what would catch a future node landing in a typo'd line ("agumonn") —
no per-node check can see that.

`scripts/import_roster.py` was deliberately NOT changed. Nothing in a sprite filename says which
family a Digimon belongs to, so the generator cannot derive `line`; a generated node now needs one
by hand before promotion into evolutions.json, the same way `"stage": null` already did. Noted in
the schema doc.

## 2026-07-19 — US-041: Build the evolution tree layout view

New `Sources/EvolutionTreeView.swift`, holding two things: `EvolutionTreeLayout`, which is pure
arithmetic (ladder-ordered columns, siblings stacked within a column, every edge resolved to the
pair of cells it joins), and `EvolutionTreeView`, which draws it. New `Tests/EvolutionTreeTests.swift`
(8). The tree is NOT on the Dex screen yet — that is US-042. It is reachable only through the
DEBUG-only `EvolutionTreeDemoView` behind `-dexTreeDemo`.

Three decisions worth not re-deriving:

- **Columns come from `ladderIndex`, not `Stage.allCases`.** Armor-Hybrid has no rung, so on a
  stage-ordered tree it is not a column at all — `allCases` would have silently filed it after
  Ultimate as an eighth stage. No shipped line has one yet; `testAnOffLadderStageIsNotGivenAColumn`
  pins the behaviour before one does, so whoever adds the first Armor form gets a failing test
  rather than a wrong column.

- **The grid is fixed-size, so connectors need no preferences.** Cell 34x34, column gap 18, row gap
  6, so cell (c,r) is at `(c*52, r*40)` and a connector is a plain `Path` between two known points.
  The alternative — flexible cells reporting their frames up through a preference before a single
  line can be drawn — is a lot of machinery for a grid whose geometry is a constant.

- **Cells are a real `HStack` of `VStack`s, NOT `.offset`.** I wrote it with `.offset` first and
  the drawing is identical to the pixel, because the sizes are fixed. What is not identical is that
  `.offset` moves a view WITHOUT moving its layout frame: every column's frame stays at the origin,
  so `ScrollViewProxy.scrollTo` cannot reach one and VoiceOver's spatial ordering has seven columns
  piled on top of each other. This cost a screenshot to notice (see below) and would have cost far
  more to notice as an accessibility bug.

Edges pointing outside the line are dropped rather than drawn to nowhere. Nothing in the shipped
roster does that, but `line` is a grouping key the evolution engine does not read, so an edge is
free to cross lines and the tree must not assume otherwise.

SIMULATOR NOTE, and the reason the `.offset` version had to be rewritten: the Agumon branch is at
columns 4-5 of 7, so a plain launch screenshots the Digitama end and the branch is off screen right.
`simctl` has no scroll command any more than it has a tap, so `-dexTreeScrollDemo` scrolls to the
last column on appear — which is what needs real column frames. The earlier gotcha still holds and
bit again: the queued notification alert survives `terminate` and `uninstall`, only `simctl erase`
clears it, and after an erase `-healthAnswered` is needed or the run lands on onboarding.

## 2026-07-19 — US-051: Add the poop model

New `Sources/Poop.swift` holding `PoopClock`, shaped deliberately as a twin of `HungerClock`: poop
is never ticked, it is recomputed from `GameState.poopUpdatedAt` against an injected clock, so a
Digimon left alone for a day is found standing in a day's worth of mess rather than in none. One
poop per **3 elapsed awake hours**, ceiling **4**.

**Elapsed time, NOT feeding — the one real design call here.** The obvious V-Pet rule is "eating
produces poop", and the AC allows either. It is wrong for this game: poop exists to be the visible
face of neglect (US-053 charges care mistakes for leaving it), and tying it to feeding would mean a
Digimon nobody ever feeds makes no mess at all — the most neglected Digimon would have the cleanest
screen. Time is the one input that keeps running while the user is not looking. Reasoning recorded
in the `PoopClock` doc comment, not just here. 3h against hunger's 4h so cleaning is its own errand
rather than always being done on the same visit as feeding; ceiling 4 matches
`HungerClock.maximumHunger` and is also what fits beside the Digimon on a 41mm screen.

**The sleep pause is a restamp, not a bank.** `advancePoop(isAsleep:now:)` treats asleep-or-dead as
paused, and a paused call accrues nothing AND moves the timestamp to `now`, so the paused stretch is
genuinely skipped instead of being paid out all at once by the next waking refresh. The honest limit,
documented on `advance`: pausing is only observed by a refresh that RUNS during it, so a night the
app slept through is still charged. Reconstructing which of the closed hours were asleep would be
guesswork — `SleepSchedule` is inferred from last night alone — so it is left as a known
approximation rather than faked. `isAsleep` is passed in (sleep is derived, not saved state, same as
`FeedAction.feed`); death is read off `healthStatus` because that one IS saved.

The part-worn-interval carry and the freeze-at-ceiling behaviour are inherited from `HungerClock`
on purpose: the frozen `poopUpdatedAt` is how long the screen has been full of mess, which US-053
gets for free rather than needing a second saved date.

`poopCountStorage`/`poopUpdatedAtStorage` added to `GameState` as OPTIONALS with computed
`poopCount`/`poopUpdatedAt` facades, for the migration reason every other storage property there
records — an optional attribute is the one shape SwiftData migrates into an already-shipped store
without a default. Accessors live in GameState.swift, not Poop.swift, because `private` is
file-scoped: an extension in another file cannot see the backing store. `init` stamps
`poopUpdatedAt` with `now` so a new game's clock starts at the hatch.

No `project.yml` edit was needed — `Sources` and `Tests` are directory globs, so both new files were
picked up by `xcodegen generate` alone (AC8 satisfied, verified by the files compiling).

Passed: build green; `Executed 592 tests, with 0 failures` (was 576, +16, all 16 confirmed by name in
an `-only-testing:DigiVPetTests/PoopClockTests -only-testing:DigiVPetTests/PoopStateTests` run).
`PoopClockTests` pins accumulation, the 3h boundary, the partial-interval carry, the ceiling, the
timestamp freeze and the pause; `PoopStateTests` drives the real `GameState` and `GameStore`,
including a drop-and-reopen for persistence.

One test is weaker than its name would suggest and is labelled as such:
`testAReopenedGameCanBeAdvancedAndSavedAgain` proves the reopen-advance-save round trip, NOT a real

## 2026-07-19 — US-052: Show poop and add the clean action

The poop model from US-051 becomes visible and cleanable, and — the part the AC does not mention but
the story is dead without — `advancePoop` is finally CALLED. It went into `refresh()` after
`updateSleepState()` and before the health read: after the sleep window, because poop is paused while
the Digimon sleeps and `isAsleep` is what says so, so running it earlier would charge the elapsed
stretch against last launch's answer; before the read, because like hunger it is owed for time
already elapsed and the read is several awaits long.

**The art is SwiftUI, and had to be.** There is no poop anywhere in the 865-file pack — every file
is a Digimon — so a `spriteFile` path would have been invented, and an invented path is a bundle
reference that fails silently as a blank square at runtime. `PoopShape` is three overlapping
ellipses in brown, widest at the base: the classic coil, and about the most that reads at 8pt.
Sized in POINTS, not sprite pixels, and deliberately without `.interpolation(.none)` — it is a
vector standing in for pixel art, so it has no pixel grid to align to and nothing to gain from
nearest-neighbour. One shape per poop rather than a count badge, because the whole point of poop in
a V-Pet is that neglect reads at a glance without anyone parsing a number.

**Placement: bottom-trailing of the sprite slot, as an overlay.** Not sharing the sprite's centre,
because the sprite is drawn with `.offset` and walks the full width of the screen — anything on its
centre line gets walked over. The corner is the one place a pile of four is always beside whatever
the Digimon is doing. Nothing is drawn at zero, so a clean screen costs no layout.

**A fifth circle forced the button diameter down from 32 to 30.** Five 32pt buttons with 6pt gaps
come to 184pt and the narrowest supported screen (41mm, watchOS 10.0 is still the deployment
target) is 176pt wide — the row would have been clipped at BOTH ends, and the 46mm simulator this
project builds against would never have shown it. 5 x 30 + 4 x 4 = 166pt, which clears 176 with a
margin at each end. `testTheRowOfFiveFitsTheNarrowestScreen` pins that arithmetic so a sixth button
or a bumped diameter fails here rather than on a small wrist.

**`clean()` restamps `poopUpdatedAt`, and that is not cosmetic.** `PoopClock` freezes the timestamp
at the instant the ceiling was reached (US-051 relies on that freeze for US-053), so a screen full
for twelve hours carries a twelve-hour-old timestamp. Zeroing the count without moving it would let
the very next `refresh()` find four intervals of elapsed time and put all four poops straight back —
cleaning visibly undoing itself. `testCleaningIsNotUndoneByTheNextRefresh` drives a real `refresh()`
after a real clean rather than asserting on the timestamp, because the visible outcome is the point.
This was the one real trap in the story.

Clean sits third, between Train and Battle: cleaning up is the third thing you do FOR the Digimon,
where Battle and the bell are what you do WITH it. `sparkles`/teal, and it holds the happy frame for
its two seconds — it is the one action whose entire reward is the Digimon being pleased. Disabled
off `poopCount`, the same value the pile is drawn from, so the button and the mess cannot disagree.

not-undone-by-refresh through the real `MainScreenModel`, refresh ageing the count, and refresh
